# 意思決定記録 - v0.2.1

## DR-001: クラウドクレデンシャル保護レベルの選定

- **ステップ**: Intent明確化
- **日時**: 2026-04-14

### 背景
GCP/Azure等のクラウドサービスクレデンシャルの保護方法を決定する必要があった。AWSはdeny-read + credential isolation（一時ファイル経由切り出し）の2層で保護しているが、新規追加サービスにも同レベルの保護を適用するか検討した。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | deny-readのみ | 実装が単純、パッチリリース範囲に収まる | 一時ファイル経由の切り出しなし |
| 2 | deny-read + credential isolation | AWS同等の完全保護 | 実装が大規模、各サービスのCLI仕様の調査が必要 |
| 3 | deny-read + 設定フレームワーク準備 | 将来の拡張に備える | フレームワーク設計のオーバーヘッド |

### 決定
**deny-readのみ** を採用

### トレードオフと判断根拠
- **得たもの**: パッチリリースとして早期に保護範囲を拡大できる
- **犠牲にしたもの**: credential isolation（エージェントにクラウドクレデンシャルを選択的に渡す機能）
- **判断根拠**: AIエージェントがクラウドCLIを直接使う場面は限定的であり、deny-readによる読み取り拒否で十分な保護を得られる。credential isolationは将来のサイクルで必要に応じて検討する

## DR-002: deny-read対象サービスの範囲

- **ステップ**: Intent明確化
- **日時**: 2026-04-14

### 背景
deny-readに追加するクラウドサービス・ツールの範囲を決定する必要があった。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | GCP + Azure のみ | 変更が最小限 | 主要ツールが保護されない |
| 2 | GCP + Azure + その他主要 | バランスの良いカバレッジ | 保護漏れの可能性 |
| 3 | 全般的にカバー | 網羅的な保護 | パスリストの肥大化 |

### 決定
**全般的にカバー** を採用（17パス）

### トレードオフと判断根拠
- **得たもの**: 主要クラウド/ツールのクレデンシャルを網羅的に保護
- **犠牲にしたもの**: パスリストの増加（4→21パス）
- **判断根拠**: deny-readの追加コストは低く（パスリストへの追記のみ）、網羅的に保護するメリットが大きい。選定基準を「認証トークン/APIキーを含むパス」と明確化し、17パスに確定した

## DR-003: Keychain制限アプローチ

- **ステップ**: Intent明確化
- **日時**: 2026-04-14

### 背景
macOS SeatbeltプロファイルでのKeychainアクセス制限の方針を決定する必要があった。現在はSecurityServer全面許可の状態。

### 選択肢

| # | 選択肢 | メリット | デメリット |
|---|--------|---------|-----------|
| 1 | Keychainアクセスの制限強化 | 不要なアイテムへのアクセスを排除 | Seatbeltの技術的制約の調査が必要 |
| 2 | Claude Code固有のプロファイル追加 | エージェント別の最適化 | エージェント数に比例して複雑化 |
| 3 | 別の意図 | - | - |

### 決定
**Keychainアクセスの制限強化** を採用

### トレードオフと判断根拠
- **得たもの**: エージェント非依存の汎用的なKeychain保護
- **犠牲にしたもの**: エージェント固有の最適化（各エージェントに必要なKeychain項目の個別許可）
- **判断根拠**: jailrunは特定エージェントに依存しない汎用ラッパーであり、エージェント別プロファイルよりも汎用的な制限強化が設計方針に合致する。技術的な実現可否は調査Unit（Unit 002）で検証する

## DR-004: Keychain制限の実装方式（Unit 002調査結果）

- **ステップ**: Construction Phase - Unit 002 技術スパイク
- **日時**: 2026-04-14

### 背景
Unit 002の技術スパイクで、macOS SeatbeltにおけるKeychain制御の実現可能な粒度を調査した。3つの戦略候補の中から最適な方式を選定する必要があった。

### 調査結果サマリ

| 戦略 | 結果 | 理由 |
|------|------|------|
| `seatbelt-item-filtering` | **不採用** | `keychain-access-*` オペレーションはmacOS 26.3で「unbound variable」。Seatbelt APIに存在しない |
| `keychain-db-regex-write-scope` | **採用** | `regex` でKeychain DBファイル + SQLite WALジャーナルへの書き込みスコープ絞り込みが有効 |
| `keep-broad-access` | **不採用** | regexスコープ絞り込みが実用的に機能するため不要 |

### 決定
**`keychain-db-regex-write-scope`** を採用

### 実装方式
- `~/Library/Keychains` の `subpath` 書き込み許可を削除
- 代わりに regex パターン `^$HOME/Library/Keychains/(login|metadata)\.keychain-db` を使用（末尾アンカーなし）
- SQLite WALモードの `-shm`/`-wal` およびSecurityServerが使用するその他の関連ファイルにマッチ
- strict regex `(-wal|-shm)?$` は実機検証で失敗したため不採用。SecurityServerが `-wal`/`-shm` 以外のファイルにも書き込む
- UUID配下のファイル（`keychain-2.db`, `TrustedPeersHelper.db`等）は許可対象外（AIエージェント不要）

### トレードオフと判断根拠
- **得たもの**: Keychainディレクトリ全体ではなく、`login.keychain-db` と `metadata.keychain-db`（+ジャーナル）のみに書き込みを制限。Data Protectionキーチェーン等の不要なファイルへの書き込みをブロック
- **犠牲にしたもの**: アイテム単位の制御（`keychain-access-*`が利用不可のため不可能）
- **判断根拠**: regex方式でClaude Codeの`security add-generic-password`が正常動作することを実機検証で確認。TLS証明書検証は`/Library/Keychains/System.keychain`を使用し影響なし
